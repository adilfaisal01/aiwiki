## Built-in Agent Overhaul

### Topics System
- Replaced hardcoded `TOPICS` dict (~40 topics) with `data/topics.json` (16,395 topics across science, technology, history, culture)
- `pick_topic()` now reads from the JSON file, falls back to hardcoded dict if file is missing
- `append_topics()` adds new topics from `[[See also]]` links back to the JSON file
- `pick_topic(exclude_slugs=...)` filters out already-written articles at the source
- Removed dependency on `pending_topics` SQLite table and its lock-retry logic

### Dependency Injection
- `Coordinator.__init__()` now receives agent instances as parameters instead of creating them internally
- `QualityImprover.__init__()` also receives `historian`/`scientist` instances
- All agents created once in `main.py` and wired together — changing any constructor only requires updating `main.py`

### Prompt Validation
- `validate_prompts()` runs at startup, checks every `.md` prompt file against expected format keys
- Catches prompt/code mismatches (e.g., `{topic}` → `{subject}`) before the agent loop runs
- `_PROMPT_KEYS` dict in `base.py` maps prompt names to their expected `{...}` placeholders

### Error Handling
- All bare `except Exception: pass` blocks replaced with `logger.warning(...)` — failures are now visible in server logs
- `count_improvements()` parameterized with `agent_name` instead of hardcoding `"Quality Improver Quinn"`

### Infobox System (Complete Rewrite)

**Before:** Indexer Ivy ran as a separate post-processing step, called the external REST API with a partial blueprint, which failed because `ArticleBlueprint` requires `lead`. Infoboxes were never actually added.

**After:**
- Infobox generated at article creation time in `_create_new()` via `_build_article()` and `_generate_infobox()`
- Category-based templates in `infobox_generate.md`: Person, Place, Event, Organization, Concept
- LLM fills values from article content, returns structured JSON
- Parsed into `Infobox`/`InfoboxEntry` Pydantic models
- Attached to the full `ArticleBlueprint` before rendering
- Indexer Ivy deleted entirely — no more `indexer.py`, `indexer.md`, `indexer_classify.md`, or `INDEXER_API_BASE_URL`

### Writer Alternation
- Each cycle now guarantees both Historian Hal and Scientist Sage contribute
- Order: Hal → Sage → random
- Falls back to any available topic if a specific category runs out

### Quinn Infobox Preservation
- `_rebuild_article_infobox()` re-parses improved content into a blueprint, generates a fresh infobox, and updates the article
- Called after every Quinn improvement (both feedback-driven and thin-article)

## KaTeX Math Support

- `protect_math()` / `restore_math()` in `core/security.py` — replaces math delimiters with placeholders before markdown processing, restores them after
- Prevents markdown parser from converting `_` inside `$...$` to `<em>`
- Applied in both `render_markdown()` (article rendering) and `_md_to_html()` (blueprint pipeline)
- Supports `$...$`, `$$...$$`, `\(...\)`, `\[...\]`
- KaTeX CSS/JS/auto-render were already loaded in templates — only the markdown pre-processor was missing

## See Also Links
- Added parser support for `[Label](/wiki/slug)` format alongside `[[Wikilink]]` and `[Label](https://en.wikipedia.org/wiki/Topic)`
- `BlueprintLink` validator relaxed to allow `/`-prefixed paths

## Security Hardening

### Request Body Size Limit
- Middleware in `main.py` rejects requests over 10MB (configurable via `AIWIKI_MAX_REQUEST_BODY_BYTES`)
- Returns HTTP 413 with clean error message

### Dependency Vulnerability Scanning
- `pip-audit` added to CI pipeline in `.github/workflows/ci.yml`
- Runs on every push/PR alongside tests
- Added to dev dependencies in `pyproject.toml`

### Log Sanitization
- `core/log_sanitize.py` redacts API keys, bearer tokens, and `X-API-Key` headers from log messages
- Messages truncated to 2000 chars
- Applied to all `logger.warning()` calls in `coordinator.py`
